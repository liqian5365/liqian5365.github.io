Context
1.什么时候使用Context
对组件树来讲，是全局数据的时候，多层传递props会变得非常的麻烦，有时候需要层层传递
使用Context可以避免通过中间元素传递props
可以让我们无需明确地传遍每一个组件，就能将值深入的穿的进组件树
Context主要应用场景在于很多不同层级的组件需要访问同样的一些数据，但是需要谨慎使用，因为这会使得组件的复用性变差

BFC原理
Block Formatting Contexts 块级格式化上下文
可以理解为一个完全封闭的盒子，盒子内部的元素无论如何摆放，也不会影响到盒子外面。只要元素触发以下任意一条，就视其具有BFC特性。
body 根元素
浮动元素： float除none以外的值
绝对定位元素：position为（absolute || fixed）
display为(inline-block || table-cell || flex)
overflow 除了 visible以为的值(hidden || auto || scroll)

深拷贝实现：

首先在全局作用域下使用 let 和 const 声明变量，变量并不会被挂载到 window 上，这一点就和 var 声明有了区别。
提升存在的根本原因就是为了解决函数间互相调用的情况
函数提升优先于变量提升，函数提升会把整个函数挪到作用域顶部，变量提升只会把声明挪到作用域顶部
var 存在提升，我们能在声明之前使用。let、const 因为暂时性死区的原因，不能在声明前使用
var 在全局作用域下声明变量会导致变量挂载在 window 上，其他两者不会

let 和 const 作用基本一致，但是后者声明的变量不能再次赋值

Mixin带来的危害：
1、Mixin可能会相互依赖，相互耦合，不利于代码维护。
2、不同的Mixin中的方法可能会互相冲突
3、Mixin非常多的时候，组件是可以感知到的，甚至还要为其做相关的处理，这样会给代码造成滚雪球式的复杂性。
React现在已经不推荐使用Mixin来解决代码复用问题。因为Mixin带来的危害比他产生的价值还要巨大。
并且React推荐使用高阶组件来代替它。

装饰模式： 在不改变原有对象的基础之下给原有对象扩展功能，是对继承关系的一种替换方案，装饰模式也叫包装器模式。decoration
目的：扩展原有对象的功能，不改变原有对象，当然在怼原有对象不改变的前提下扩展功能有很多种做法：
1继承 2 动态代理 3装饰器模式等等
动态代理是最好的一种解决方案。

HOC: Higher-Order Components
高阶组件就是一个函数，且该函数接受一个组件作为参数，并返回一个新的组件。

HOC可以实现的功能：
1、组合渲染：可以使用任何其他嘴贱和原组件进行组合渲染，达到样式，布局复用等效果。
2、条件渲染：根据特定的属性决定原组件是否渲染
3、操作props：可以对传入组建的props进行增加，修改，删除或者根据特定的props进行特殊的操作
4、获取refs
5、渲染劫持，高阶组件在render函数中做非常多的操作，从而控制原组件的渲染输出，只要改变了原组件的渲染都成为渲染劫持

React.cloneElement()克隆并返回一个新的react元素，使用element作为起点，生成的元素将会拥有原始元素props与新props的前合并。
新的子级会替换现有的子级，来自原始元素的key和ref将会保留。

高阶组件就是一个没有副作用的纯函数。各个高阶组件不会互相依赖耦合。
高阶组件也有可能造成冲突，但我们可以在遵守约定的情况下避免这些行为。
高阶组件并不关心数据使用的方式和原因，而被包裹的组件也不关心数据来自何处，高阶组件的增加不会为原组件增加负担。
缺陷：
HOC需要在原组件上进行包裹或者嵌套。大量使用会产生非常多的嵌套，让调用变得复杂
HOC可以劫持props,在不遵守约定的情况下也可以造成冲突。

函数式编程
纯函数：没有副作用的函数，相同的数组有相同的输出
仅取决于提供的输入，而不依赖于任何在函数求职或调用间隔时可能变化的隐藏状态和外部状态
不会造成超出作用域的变化，比如修改全局变量或者引用传递的参数
使用纯函数的代码绝不会更改或破坏全局状态，有助于提高代码的可测试性和可维护性
函数式变成采用声明式风格，易于推理，提高代码的可读性
函数式编程将函数视为积木，通过一等高级函数来提高代码模块化和可重用性
可以利用响应式编程组合各个函数来降低事件驱动程序的复杂性
